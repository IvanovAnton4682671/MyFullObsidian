#DRY
___
## Don’t Repeat Yourself («Не повторяйся»)
### Формулировка принципа

Принцип DRY был сформулирован Энди Хантом и Дэйвом Томасом в их классической книге **«Программист-прагматик»** (_The Pragmatic Programmer: Your Journey to Mastery_, 1999). Это один из самых известных и часто цитируемых принципов в разработке ПО

> _«Каждая часть знания должна иметь единственное, непротиворечивое, авторитетное представление в системе»_  
> Andy Hunt, Dave Thomas. _The Pragmatic Programmer_ (1999)

Авторы подчёркивают, что DRY - это не просто про дублирование кода. Это про дублирование **знаний** (knowledge), **намерений** (intent) и **логики**. Два идентичных фрагмента кода могут не нарушать DRY, если они представляют разные знания, которые просто совпали синтаксически. И наоборот - два внешне разных фрагмента могут нарушать DRY, если реализуют одно и то же знание разными способами

### Простое объяснение «на пальцах»

Представьте, что вы часто переезжаете и указываете свой контактный телефон в десятке разных документов: в договоре аренды, в банковских анкетах, в регистратуре поликлиники, в подписке на интернет, у работодателя, в страховке, в ветеринарной клинике и на сайте знакомств

Всё идёт хорошо, пока вы не меняете номер. Теперь вам нужно обновить его в десяти независимых местах. Если вы забудете про одно, начнутся проблемы: банк не дозвонится, врач не предупредит о сдаче анализов, потенциальный партнёр решит, что вы его игнорируете. Вы становитесь заложником собственной системы, в которой **одно знание (ваш телефонный номер) дублируется во многих местах**

Принцип DRY предлагает альтернативу: завести **единый источник истины** (Single Source of Truth) - например, персональную визитку с QR-кодом, ведущим на страницу, где ваш номер всегда актуален. Меняете номер в одном месте - обновляется везде

В программировании это означает: если бизнес-правило «скидка для постоянных клиентов = 5%» встречается в пяти разных местах кода, то при изменении скидки до 7% вам придётся искать и править все пять мест. Вы почти гарантированно пропустите одно, и в системе поселится баг. DRY требует выделить это правило в одно место - константу, метод, модуль - и ссылаться на него отовсюду

### Реальный пример
#### Нарушение DRY: разбросанная логика

Рассмотрим простое приложение для интернет-магазина, где в разных частях кода рассчитывается стоимость доставки. Правило бизнеса одно: «Бесплатная доставка при сумме заказа от 5000 рублей, иначе - 300 рублей»

Разработчик реализовал это правило в нескольких местах:

```csharp
// В контроллере корзины
public class CartController
{
    public decimal GetShippingCost(Cart cart)
    {
        if (cart.TotalAmount >= 5000)
        {
            return 0;
        }
        else
        {
            return 300;
        }
    }
}

// В сервисе оформления заказа
public class OrderService
{
    public decimal GetShippingCost(Cart cart)
    {
        decimal shippingCost;
        if (cart.TotalAmount >= 5000)
        {
            shippingCost = 0;
        }
        else
        {
            shippingCost = 300;
        }
        return shippingCost;
    }
}

// В генераторе отчётов
public class ReportGenerator
{
    public string GenerateShippingReport(Cart cart)
    {
        decimal shipping = cart.TotalAmount >= 5000 ? 0 : 300;
        return $"Shipping cost: {shipping}";
    }
}
```

**Проблема:** Когда бизнес решит изменить порог бесплатной доставки на 7000 рублей и стоимость платной доставки на 350 рублей, разработчику придётся найти **все** места, где закодировано это правило. В реальном проекте таких мест могут быть десятки - контроллеры, сервисы, стратегии расчёта налогов, email-шаблоны. Пропуск хотя бы одного приведёт к рассинхронизации поведения системы

Это классическое нарушение DRY: **знание «правило расчёта стоимости доставки» представлено в системе многократно, не имея единого авторитетного источника**

#### Применяем DRY: Единый источник истины

Выделяем правило в один метод (или класс), который становится единственным местом, где определено это знание:

```csharp
public static class ShippingCalculator
{
    private const decimal FreeShippingThreshold = 5000m;
    private const decimal StandardShippingCost = 300m;
	
    public static decimal Calculate(Cart cart)
    {
        return cart.TotalAmount >= FreeShippingThreshold ? 0 : StandardShippingCost;
    }
}
```

Теперь весь остальной код **ссылается** на этот единый источник:

```csharp
public class CartController
{
    public decimal GetShippingCost(Cart cart)
    {
        return ShippingCalculator.Calculate(cart);
    }
}

public class OrderService
{
    public decimal GetShippingCost(Cart cart)
    {
        return ShippingCalculator.Calculate(cart);
    }
}

public class ReportGenerator
{
    public string GenerateShippingReport(Cart cart)
    {
        decimal shipping = ShippingCalculator.Calculate(cart);
        return $"Shipping cost: {shipping}";
    }
}
```

При изменении порога или стоимости доставки мы правим **только константы в `ShippingCalculator`** - и поведение всей системы меняется согласованно. Это и есть DRY в действии

#### Пример скрытого нарушения DRY

Авторы «Программиста-прагматика» предупреждают: дублирование может быть не только в коде, но и в процессах. Представьте, что у нас есть два метода:

```csharp
public void ProcessRefund(Order order)
{
    // 1. Проверить статус заказа (одинаково)
    if (order.Status != OrderStatus.Completed)
        throw new InvalidOperationException("Refund only allowed for completed orders");
    // 2. Рассчитать сумму к возврату (одинаково)
    decimal refundAmount = CalculateRefundAmount(order);
    // 3. Выполнить возврат (разное)
    ExecuteRefund(order, refundAmount);
}

public void ProcessPartialRefund(Order order, decimal amount)
{
    // 1. Проверить статус заказа (одинаково)
    if (order.Status != OrderStatus.Completed)
        throw new InvalidOperationException("Refund only allowed for completed orders");
    // 2. Рассчитать сумму к возврату (одинаково? или нет?)
    decimal maxRefund = CalculateRefundAmount(order);
    if (amount > maxRefund)
        throw new InvalidOperationException("Partial refund exceeds allowed amount");
    // 3. Выполнить возврат (разное)
    ExecutePartialRefund(order, amount);
}
```

Проверка статуса заказа «Refund only allowed for completed orders» и текст исключения дублируются. Если статусная модель изменится (добавится `Cancelled` с правом возврата), придётся править оба метода. Более того, сам текст исключения - это тоже знание. DRY-решение - вынести валидацию в отдельный метод `EnsureRefundAllowed(order)`

### Зачем нужен принцип DRY?

1. **Упрощение внесения изменений**
   Когда знание представлено в единственном месте, изменение этого знания требует правки ровно одной точки. Это ускоряет разработку и снижает риск ошибок
2. **Предотвращение рассинхронизации**
   Если знание размножено, рано или поздно разные копии начнут различаться - из-за неполного рефакторинга, ошибок при слиянии веток, забывчивости. Система становится противоречивой
3. **Повышение надёжности**
   Единый, протестированный модуль, отвечающий за конкретное знание, используется везде. Вероятность ошибки снижается: мы тестируем один раз, а не N раз в каждом контексте
4. **Улучшение читаемости**
   Когда в коде встречается вызов `ShippingCalculator.Calculate(cart)`, намерение очевидно. Когда же разбросаны магические числа `5000` и `300`, читателю приходится догадываться, знание ли это одного правила или случайное совпадение
5. **Облегчение рефакторинга**
   Собрать разбросанное знание в единый модуль - это первый шаг к улучшению архитектуры (например, вынесение в отдельный сервис, конфигурирование через БД)

### Как применять DRY на практике?

1. **Отличайте дублирование знаний от дублирования кода**
   Два куска кода могут быть идентичны синтаксически, но представлять разные знания, которые просто совпали. Если их объединить в одну абстракцию, то при изменении одного из знаний сломается другое. Это называется **преждевременной абстракцией** (premature abstraction), которая опаснее дублирования
   **Пример:**
   В одном месте `decimal discount = price * 0.10;` - это «скидка 10% за оптовую покупку». В другом месте `decimal tax = price * 0.10;` - это «налог 10%». Код идентичен, знание разное. Объединение в метод `ApplyTenPercent(price)` приведёт к катастрофе при изменении ставки налога до 12%
2. **Ищите дублирование в процессах, а не только в коде**
	- Дублирование в документации и комментариях: комментарий, описывающий работу метода, может устареть, если метод изменился. Лучше сделать код самодокументированным
	- Дублирование в тестах: если тесты повторяют логику продакшен-кода, при изменении этой логики придётся править и код, и тесты
	- Дублирование в конфигурациях: одинаковые строки подключения в нескольких конфигах - тоже нарушение DRY
3. **Выносите дублирующееся знание в подходящую абстракцию**
	- **Метод** - для повторяющейся последовательности действий
	- **Класс/Сервис** - для набора связанных методов, оперирующих общим знанием
	- **Константа/Enum** - для магических чисел и строк, представляющих бизнес-значения
	- **Библиотека/Пакет** - для общей функциональности, используемой несколькими проектами
4. **Следуйте «Правилу трёх» (Rule of Three)**
   Не выносите дублирование в абстракцию при первом же повторе. Дождитесь третьего появления. Только когда один и тот же паттерн проявляется в трёх местах, становится понятно, как правильно абстрагировать знание, не создавая лишней сложности
5. **Используйте рефакторинг «Извлечение метода» (Extract Method)**
   Самый простой и частый способ устранения дублирования. Если вы видите, что несколько строк кода повторяются в разных местах, выделите их в метод с осмысленным названием
6. **Соблюдайте баланс между DRY и KISS**
   Иногда устранение дублирования приводит к созданию сложной иерархии классов, которую трудно понимать. В таких случаях небольшое контролируемое дублирование может быть проще и дешевле в поддержке, чем сложная абстракция. Всегда взвешивайте: снижает ли абстракция суммарную сложность системы?
7. **Применяйте «DRY наоборот» для защиты от преждевременной связи**
   Если два модуля эволюционируют независимо, даже если сегодня они выглядят одинаково, не связывайте их общей абстракцией. Они должны оставаться независимыми, чтобы изменения в одном не затрагивали другой. Это особенно актуально в микросервисной архитектуре и при разделении ограниченных контекстов (Bounded Contexts) в DDD

### Связь с другими принципами

- **KISS (Keep It Simple, Stupid)**: DRY и KISS могут входить в противоречие. Устранение дублирования иногда требует введения дополнительных абстракций (интерфейсов, базовых классов, стратегий), которые усложняют код. Прагматичный подход: применяй DRY там, где это **упрощает** систему в целом. Если абстракция для устранения дублирования усложняет понимание, возможно, лучше оставить небольшое дублирование
- **YAGNI (You Ain't Gonna Need It)**: Преждевременное создание общей абстракции «на будущее», когда дублирование ещё не проявилось, нарушает YAGNI. DRY включается тогда, когда дублирование **уже существует** и причиняет боль
- **SRP (Single Responsibility Principle)**: Выделяя дублирующееся знание в отдельный модуль, мы естественно создаём класс с единственной ответственностью. `ShippingCalculator` отвечает только за расчёт доставки - это SRP в чистом виде
- **OCP (Open/Closed Principle)**: Единый модуль, содержащий знание, легче расширить (поддержать новое правило доставки), не модифицируя всех клиентов этого знания
- **ISP (Interface Segregation Principle)**: DRY, применённый к интерфейсам, означает, что не следует заставлять клиента зависеть от методов, которые он не использует. Общий «толстый» интерфейс для разного контекста - это тоже форма вредного дублирования обязанностей

### Основные источники

- **Hunt A., Thomas D.** _The Pragmatic Programmer: Your Journey to Mastery_ (1999) - первоисточник принципа DRY, глава «The Evils of Duplication», где подробно разбираются категории дублирования
- **Thomas D., Hunt A.** _The Pragmatic Programmer: 20th Anniversary Edition_ (2019) - обновлённое издание с современными примерами и уточнением принципа для микросервисов и облачной разработки
- **Fowler M.** _Refactoring: Improving the Design of Existing Code_ (2nd Edition, 2018) - практические техники устранения дублирования (Extract Method, Extract Class, Pull Up Method)
- **McConnell S.** _Code Complete_ (2nd Edition, 2004) - глава о качестве кода, где дублирование рассматривается как один из главных источников проблем сопровождения
