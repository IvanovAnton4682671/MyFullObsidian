#KISS
___
## Keep It Simple, Stupid («Делай проще, дурак»)
### Формулировка принципа

Принцип KISS был сформулирован в 1960-х годах инженером компании Lockheed **Кларенсом «Келли» Джонсоном** (Clarence «Kelly» Johnson), руководителем легендарного подразделения Skunk Works, создававшего передовые военные самолёты, включая SR-71 Blackbird:

> _«Keep it simple, stupid»_
> _(Делай проще, дурак)_
> Келли Джонсон (приписывается)

Полная версия принципа, которую Джонсон доносил до своих инженеров, звучала так:

> _«Наши самолёты должны быть ремонтопригодны механиком со стандартным набором инструментов в полевых условиях, в условиях боевого стресса. Если ваша конструкция сложна, значит, она никуда не годится»_

В контексте разработки программного обеспечения принцип KISS означает:

> _«Простота должна быть ключевой целью при проектировании, и следует избегать неоправданной сложности»_

Не существует единственного канонического академического определения, однако общепринятая трактовка в индустрии такова:

> _«Любая система работает лучше всего, если она проста. Сложность неизбежна, но её следует минимизировать, не жертвуя функциональностью»_

Основная идея: большинство систем работают лучше и надёжнее, если сделаны просто, а не усложнены. Простота должна быть как целью проектирования, так и критерием оценки решения

### Простое объяснение «на пальцах»

Представьте, что вам нужно построить скворечник. Вы можете:
- **Простой вариант:** несколько досок, гвозди, один вход, деревянный шест - скворечник готов
- **Сложный вариант:** дистанционно управляемая дверца, видеокамера, система подогрева с термодатчиком, автоматическая кормушка, солнечная батарея, Wi-Fi-модуль для трансляции жизни птиц на YouTube

У сложного варианта множество точек отказа: сервопривод дверцы заклинит, датчик намокнет, хакер взломает Wi-Fi и захватит скворечник. Цель-то была - дать птицам дом, а не построить «умный IoT-скворечник».

В программировании то же самое. Задача: «сохранить настройки приложения»
– **KISS-решение**: записать несколько значений в JSON-файл рядом с программой
– **Сложное решение**: развернуть сервер Redis, поднять кластер, написать уровень кеширования, добавить репликацию и веб-интерфейс для просмотра настроек

При добавлении каждого нового компонента растёт когнитивная нагрузка, вероятность ошибок, время отладки и стоимость поддержки. Принцип KISS напоминает: **делай ровно столько, сколько нужно для решения задачи, и ни строчкой больше**

### Реальный пример
#### Нарушение KISS: переусложнение простой операции

Часто разработчики, вдохновившись паттернами проектирования, применяют их без реальной необходимости, усложняя код. Рассмотрим банальную задачу: вычислить сумму двух чисел и вывести её в консоль

Переусложнённое решение «энтерпрайз-уровня»:

```csharp
// Интерфейс для стратегии математической операции
public interface IMathOperation
{
    decimal Execute(decimal a, decimal b);
}

// Конкретная реализация сложения
public class AdditionOperation : IMathOperation
{
    public decimal Execute(decimal a, decimal b)
    {
        return a + b;
    }
}

// Фабрика операций
public static class MathOperationFactory
{
    public static IMathOperation CreateOperation(string operationType)
    {
        return operationType switch
        {
            "Addition" => new AdditionOperation(),
            _ => throw new NotSupportedException($"Operation {operationType} is not supported")
        };
    }
}

// Абстракция вывода результата
public interface IOutputWriter
{
    void Write(string message);
}

public class ConsoleOutputWriter : IOutputWriter
{
    public void Write(string message)
    {
        Console.WriteLine(message);
    }
}

// Сервис-координатор
public class CalculationService
{
    private readonly IOutputWriter _outputWriter;
	
    public CalculationService(IOutputWriter outputWriter)
    {
        _outputWriter = outputWriter;
    }
	
    public void CalculateAndOutput(decimal a, decimal b, string operationType)
    {
        IMathOperation operation = MathOperationFactory.CreateOperation(operationType);
        decimal result = operation.Execute(a, b);
        string formattedMessage = $"Result: {result}";
        _outputWriter.Write(formattedMessage);
    }
}

// Использование
var outputWriter = new ConsoleOutputWriter();
var calculationService = new CalculationService(outputWriter);
calculationService.CalculateAndOutput(5, 3, "Addition");
```

**Что здесь не так?**  
- Шесть типов (два интерфейса, три класса, статическая фабрика) ради операции сложения двух чисел
- Расширяемость (новые операции, новые способы вывода) с вероятностью 99% никогда не понадобится в контексте реальной задачи
- Разработчик, который откроет этот код через полгода, потратит время, пытаясь понять, зачем здесь фабрика и интерфейсы

#### KISS-решение: простота и прямолинейность

Для задачи «сложить два числа и вывести результат» достаточно трёх строк:

```csharp
decimal a = 5m;
decimal b = 3m;
Console.WriteLine($"Result: {a + b}");
```

Этот код:
- Мгновенно понятен
- Не содержит ни одного лишнего уровня косвенности
- Задача решена оптимальным способом

Если в будущем действительно потребуется поддерживать разные операции и форматы вывода - тогда и будем рефакторить, абстрагируя стабильные точки расширения. Но **не раньше**

Простое и достаточное решение:

```csharp
public static bool IsValidEmail(string email)
{
    if (string.IsNullOrWhiteSpace(email))
        return false;
	
    try
    {
        var addr = new System.Net.Mail.MailAddress(email);
        return addr.Address == email;
    }
    catch
    {
        return false;
    }
}
```

Либо совсем простой вариант для быстрой клиентской валидации:

```csharp
public static bool IsValidEmail(string email)
{
    return !string.IsNullOrWhiteSpace(email) 
        && email.Contains('@') 
        && email.Contains('.');
}
```

### Зачем нужен принцип KISS?

1. **Снижение когнитивной нагрузки**
   Простой код быстрее читается и понимается. Разработчик, который видит код впервые, не тратит часы на распутывание хитроумных абстракций
2. **Уменьшение количества ошибок**
   Чем больше кода, тем больше потенциальных дефектов. Простые решения содержат меньше точек отказа
3. **Облегчение отладки и тестирования**
   Простую функцию с чистыми входными и выходными данными гораздо легче покрыть тестами и отладить в случае ошибки
4. **Упрощение поддержки и расширения**
   Простой код легче изменять. Если кодовая база - это переплетение абстракций, даже небольшое изменение может вызвать каскад поломок
5. **Ускорение онбординга новых разработчиков**
   Простая, прямолинейная архитектура позволяет новому члену команды быстрее стать продуктивным

### Как применять KISS на практике?

1. **Решайте конкретную задачу, а не воображаемую**
   Перед тем как написать код, спросите себя: «Усложняю ли я решение ради проблемы, которой ещё нет?» Если вы не уверены, что в будущем потребуется поддержка 10 видов отчётов, не закладывайте фабрику отчётов прямо сейчас
2. **Предпочитайте прямолинейные решения**
   Если задачу можно решить последовательностью из трёх операторов, не заворачивайте её в классы, интерфейсы и фабрики только потому, что «так правильно». Код должен быть адекватен сложности задачи
3. **Боритесь с «YAGNI-нарушениями» (You Ain't Gonna Need It)**
   Принципы KISS и YAGNI тесно связаны: не добавляйте функциональность, пока она не нужна. Каждая лишняя строчка кода - это потенциальный источник ошибок и точка отказа
4. **Избегайте «сверхинженерии» (over-engineering)**
   Не используйте паттерны проектирования только ради паттернов. Если паттерн не решает конкретную проблему сложности, он её создаёт. Сигнал опасности - когда вы ловите себя на мысли: «Сейчас я сделаю здесь стратегию, потому что у нас в каждой заметке по SOLID была стратегия»
5. **Рефакторите к простоте**
   После того как код заработал, пересмотрите его с вопросом: «Можно ли сделать проще?» Часто первая реализация содержит лишние абстракции и обобщения, которые можно безболезненно удалить
6. **Придерживайтесь правила «Одного уровня абстракции на функцию»**
   Функция, которая вперемешку вызывает низкоуровневые операции с DOM, выполняет HTTP-запросы и обрабатывает бизнес-правила, сложна для восприятия. Выносите связанные действия в отдельные методы с понятными именами. Это делает код проще, не добавляя лишней архитектуры
7. **Используйте вопрос «Зачем?»**
   При код-ревью спрашивайте: «Зачем здесь этот интерфейс?», «Почему этот класс наследуется от базового?», «Есть ли второй потребитель этой абстракции?». Если ответ - «на всякий случай» или «для расширяемости», с большой вероятностью код переусложнён

### Связь с другими принципами

- **YAGNI (You Ain't Gonna Need It)** прямо вытекает из KISS. KISS говорит «делай просто», YAGNI конкретизирует: «не реализуй то, что не требуется прямо сейчас». Вместе они формируют мощный фильтр против переусложнения
- **DRY (Don't Repeat Yourself)** иногда входит в противоречие с KISS. Неумеренное вынесение повторяющегося кода в абстракции может усложнить систему. Если два куска кода совпадают чисто случайно, объединять их не стоит - простое дублирование иногда лучше сложной общей абстракции
- **SOLID** - особенно SRP, ISP, DIP - служат инструментами для управления сложностью, но при слепом применении могут её увеличить. KISS напоминает: применяй SOLID осознанно, только в тех точках, где это снижает суммарную сложность, а не добавляет её
- **Принцип наименьшего удивления (Principle of Least Astonishment)** - простой код, как правило, ведёт себя предсказуемо и не удивляет коллег

### Основные источники

- **Johnson C. L. (Kelly Johnson)** - считается автором принципа KISS, сформулированного в Lockheed Skunk Works в 1960-х годах
- **Hunt A., Thomas D.** _The Pragmatic Programmer: Your Journey to Mastery_ (1999) - книга, популяризировавшая KISS в среде разработчиков ПО
- **McConnell S.** _Code Complete: A Practical Handbook of Software Construction_ (2nd Edition, 2004) - глава о простоте и управлении сложностью, где KISS рассматривается как ключевая цель проектирования
- **Raymond E. S.** _The Art of Unix Programming_ (2003) - изложение философии Unix, одним из правил которой является «Правило простоты»: «Проектируйте для простоты; добавляйте сложность, только когда это действительно необходимо»
