#YAGNI
___
## You Ain’t Gonna Need It («Тебе это не понадобится»)
### Формулировка принципа

Принцип YAGNI возник в конце 1990-х годов в сообществе экстремального программирования (Extreme Programming, XP) и был сформулирован **Кентом Беком** (Kent Beck) и **Роном Джеффрисом** (Ron Jeffries). Это одна из ключевых практик XP, направленная на борьбу с излишней функциональностью и преждевременным усложнением

> _«Всегда реализуйте вещи тогда, когда они вам действительно нужны, а не тогда, когда вы просто предвидите, что они вам понадобятся»_  
> Kent Beck. _Extreme Programming Explained: Embrace Change_ (1999)

Лаконичная и часто цитируемая формулировка:

> _«You Ain’t Gonna Need It» (YAGNI)_  
> _«Тебе это не понадобится»_

Ключевой посыл: **не добавляйте функциональность «про запас»**. Каждая строчка кода, которая не решает текущую задачу, - это багаж, который нужно писать, отлаживать, тестировать, документировать и поддерживать. Вероятность того, что «заложенная на будущее» возможность будет использована именно в том виде, как её спроектировали, ничтожно мала. При этом затраты на её создание и сопровождение - реальны и ощутимы

### Простое объяснение «на пальцах»

Представьте, что вы собираетесь в командировку на два дня. Список реально необходимых вещей: ноутбук, зарядка, сменная рубашка, зубная щётка. Но внутренний голос нашёптывает:
- А вдруг пойдёт дождь? Возьму зонт
- А вдруг там будет бассейн? Возьму плавки
- А вдруг пригласят на официальный ужин? Возьму пиджак
- А вдруг будет холодно? Возьму свитер
- А вдруг задержусь на неделю? Возьму ещё трое брюк
- А вдруг понадобится что-то починить? Возьму набор отвёрток

И вот уже вместо лёгкого рюкзака вы тащите неподъёмный чемодан. Большинство вещей не понадобятся (в командировке на два дня плавки для бассейна - это фантазия), но вы платите за них усилием: тащите, распаковываете, разыскиваете нужное среди кучи хлама, боитесь забыть в отеле

YAGNI в разработке ПО - это дисциплина: **не класть в рюкзак вещи, которые не нужны прямо сейчас, как бы соблазнительно ни было «предусмотреть всё»**

Заказчик сегодня запросил кнопку «Сохранить в PDF». Не нужно реализовывать «на будущее» целую подсистему экспорта, поддерживающую PDF, DOCX, XLSX, CSV и JSON через фабрику экспортёров. Сделайте PDF. Когда завтра заказчик скажет: «А теперь ещё и в Excel», - вот тогда и спроектируете расширение, имея на руках реальные требования, а не фантазии

### Реальный пример

Допустим, тимлид или архитектор говорит: «Нам нужен компонент для отправки уведомлений. Давайте сразу сделаем поддержку Email, SMS, Telegram и Push, потому что в будущем точно пригодится. Я почитал roadmap на два года вперёд - там такое планируется». Команда реализует систему с фабриками, стратегиями и конфигурациями:

```csharp
public enum NotificationType
{
    Email,
    Sms,
    Telegram,
    Push
}

// Интерфейс для стратегии отправки
public interface INotificationSender
{
    void Send(string recipient, string message);
}

// Четыре реализации «на будущее»
public class EmailSender : INotificationSender
{
    public void Send(string email, string message)
    {
        // Реальная реализация SMTP-отправки
        Console.WriteLine($"Email sent to {email}: {message}");
    }
}

public class SmsSender : INotificationSender
{
    public void Send(string phone, string message)
    {
        Console.WriteLine($"SMS sent to {phone}: {message}");
    }
}

public class TelegramSender : INotificationSender
{
    public void Send(string chatId, string message)
    {
        Console.WriteLine($"Telegram message sent to {chatId}: {message}");
    }
}

public class PushSender : INotificationSender
{
    public void Send(string deviceToken, string message)
    {
        Console.WriteLine($"Push notification sent to device {deviceToken}: {message}");
    }
}

// Фабрика для создания отправителя по типу
public static class NotificationSenderFactory
{
    public static INotificationSender Create(NotificationType type)
    {
        return type switch
        {
            NotificationType.Email => new EmailSender(),
            NotificationType.Sms => new SmsSender(),
            NotificationType.Telegram => new TelegramSender(),
            NotificationType.Push => new PushSender(),
            _ => throw new ArgumentOutOfRangeException(nameof(type))
        };
    }
}

// Сервис, который оркестрирует отправку
public class NotificationService
{
    public void Notify(string recipient, string message, NotificationType type)
    {
        var sender = NotificationSenderFactory.Create(type);
        sender.Send(recipient, message);
    }
}
```

**Что пошло не так:**
- На момент написания кода бизнесу **реально нужен только Email**. Никаких требований по SMS, Telegram, Push не поступало
- Разработчики потратили время на реализацию трёх классов-заглушек и фабрики, которая не тестировалась в реальных условиях
- Если через год действительно потребуется SMS, его спецификация, скорее всего, будет отличаться от выдуманной сегодня. Код всё равно придётся переписывать
- Увеличилась когнитивная нагрузка: любой разработчик, читающий этот модуль, видит четыре класса и гадает, какие из них реально используются
- Заглушки падают, если их случайно вызвать (например, через неправильную конфигурацию), или просто занимают место

#### YAGNI-решение: только то, что нужно сейчас

Прагматичный подход: реализуем Email-отправку, потому что это требование текущей итерации:

```csharp
public class EmailSender
{
    public void Send(string email, string message)
    {
        Console.WriteLine($"Email sent to {email}: {message}");
    }
}
```

Никаких интерфейсов, фабрик и `switch` - просто один класс, решающий конкретную задачу. Код минимален, сразу понятен, тестируется без моков

#### Что делать, когда SMS всё-таки понадобились?

Через несколько месяцев появляется реальное требование: «Добавить отправку SMS для двухфакторной аутентификации». **Вот теперь** у нас есть конкретный юзкейс, и мы можем осмысленно провести рефакторинг:
1. Выделить интерфейс `INotificationSender` с методом `Send`
2. Переименовать `EmailSender` в `EmailNotificationSender : INotificationSender`
3. Создать `SmsNotificationSender : INotificationSender`
4. Внедрить зависимость от `INotificationSender` в нужный сервис

Теперь абстракция оправдана, поскольку у нас **два реальных потребителя**, и мы понимаем, как они оба должны работать. До этого момента абстракция была преждевременной и нарушала YAGNI

### Зачем нужен принцип YAGNI?

1. **Экономия времени разработки**
   Каждая минута, потраченная на ненужную сейчас функцию, украдена у функций, которые реально важны для продукта прямо сейчас. Отложенная работа накапливает долг
2. **Снижение сложности кодовой базы**
   Неиспользуемая функциональность загромождает код, делает его запутанным. Разработчики тратят время, разбираясь, где «живой» код, а где - выдуманный запас
3. **Предотвращение ложной расширяемости**
   «Гибкая архитектура», построенная на предположениях, часто оказывается гибкой не в ту сторону. Изменение реальных требований потребует переписывания этой архитектуры, а значит, усилия были потрачены зря
4. **Облегчение тестирования**
   Меньше кода - меньше тестов. Неиспользуемые методы всё равно нужно тестировать (или мучительно решать, стоит ли), иначе метрики покрытия падают. YAGNI избавляет от этой дилеммы
5. **Упрощение принятия решений**
   Когда разработчик говорит: «Я добавлю этот параметр на будущее» или «Сделаю универсальную фабрику», YAGNI останавливает его: «Подожди, тебе это точно нужно сейчас?». Это упрощает планирование и снижает риск золотой пилюли

### Как применять YAGNI на практике?

1. **Добавляйте фичу только тогда, когда есть конкретное требование**
   Требование должно быть выражено в пользовательской истории, зафиксировано в баг-трекере или обсуждено с заказчиком. Если звучит фраза «Возможно, когда-нибудь…», это красный флаг. Остановитесь
2. **Боритесь с «архитектурным зудом»**
   Разработчикам нравится проектировать красивые универсальные системы. Это приятно, но вредно для продукта. Прагматик умеет отличать профессиональное удовольствие от реальной потребности бизнеса. Направьте творческую энергию на упрощение, а не на усложнение
3. **Следуйте коротким итерациям**
   В Agile-процессах требования уточняются от итерации к итерации. Если вы реализуете то, что нужно в ближайшие две недели, а не «глобальное видение на год», вы естественно следуете YAGNI
4. **Применяйте рефакторинг по мере необходимости**
   YAGNI не запрещает рефакторинг! Когда появляется второе требование, проведите рефакторинг к более общей абстракции. Делайте это тогда, когда у вас есть **два и более реальных примера**, а не на основе одной фантазии. Как советовал Мартин Фаулер: «Рефакторируйте, когда боль от дублирования становится ощутимой»
5. **Отказывайтесь от «спекулятивного кода» на код-ревью**
   Если на ревью вы видите метод, интерфейс или параметр, которые не используются и были добавлены «на всякий случай», задайте вопрос: «Какую конкретную задачу это решает сейчас?» Если ответ расплывчатый - смело убирайте
6. **Признавайте, что предсказания будущего ошибочны**
   Индустрия десятилетиями доказывает: мы очень плохо умеем предсказывать, как будет развиваться программный продукт. Бизнес-требования меняются быстрее, чем архитектурные фантазии успевают окупиться. Проще и дешевле реагировать на реальность, чем пытаться её предугадать

### Связь с другими принципами

- **KISS (Keep It Simple, Stupid)**: YAGNI - это конкретизация KISS во временно́й плоскости. KISS говорит «делай просто сейчас», YAGNI добавляет: «и не усложняй ради воображаемого завтра». Вместе они - два главных оружия против over-engineering
- **DRY (Don't Repeat Yourself)**: YAGNI помогает избежать **преждевременной абстракции** ради DRY до того, как дублирование стало реальной проблемой. Правило: «Не выноси общий код, пока у тебя не появилось трёх реальных случаев дублирования». Первые два могут быть совпадением
- **SRP (Single Responsibility Principle)** из SOLID: YAGNI помогает соблюдать SRP, уберегая от закладывания в один класс множества ответственностей «на вырост». Класс, который делает только то, что нужно сейчас, обычно имеет одну ответственность
- **OCP (Open/Closed Principle)**: YAGNI и OCP иногда конфликтуют. OCP призывает проектировать расширяемость заранее, YAGNI - отложить расширяемость до реальной необходимости. Компромисс: применяй OCP только в тех точках, где изменения уже происходили неоднократно и паттерн изменений понятен. Если ты не знаешь паттерна - YAGNI побеждает
- **BDUF (Big Design Up Front)**: YAGNI - это прямая противоположность практике «большого проектирования наперёд». Экстремальное программирование утверждает, что проектирование должно быть эволюционным, а не всеобъемлющим на старте

### Основные источники

- **Beck K.** _Extreme Programming Explained: Embrace Change_ (1st ed., 1999, 2nd ed., 2004) - первоисточник, где YAGNI вводится как одна из фундаментальных практик XP
- **Jeffries R., Anderson A., Hendrickson C.** _Extreme Programming Installed_ (2001) - подробное описание практик XP, включая YAGNI, от соавтора и идеолога экстремального программирования
- **Fowler M.** _Refactoring: Improving the Design of Existing Code_ (2nd Edition, 2018) - концепция эволюционного проектирования и рефакторинга как альтернативы «закладыванию на будущее»
- **Hunt A., Thomas D.** _The Pragmatic Programmer: Your Journey to Mastery_ (1999, 20th Anniversary Ed. 2019) - в главах о «Pragmatic Philosophy» авторы неоднократно обращаются к идее YAGNI как к защите от спекулятивного программирования
- **Poppendieck M., Poppendieck T.** _Lean Software Development: An Agile Toolkit_ (2003) - принцип «Решай, как можно позже» (Decide as Late as Possible) из бережливого производства, тесно связанный с YAGNI
