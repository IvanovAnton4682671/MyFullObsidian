## Liskov Substitution Principle (LSP)
### Формулировка принципа

Принцип назван в честь **Барбары Лисков** (Barbara Liskov), впервые сформулировавшей его в 1987 году на конференции «Data Abstraction and Hierarchy». Математически точное определение звучит так:

> _«Пусть q(x) - свойство, доказуемое для объектов x типа T. Тогда q(y) должно быть доказуемо для объектов y типа S, где S является подтипом T»_
> Barbara Liskov, _Data Abstraction and Hierarchy_ (1987)

Роберт Мартин переложил этот принцип на объектно-ориентированное проектирование в более доступной форме:

> _«Функции, которые используют ссылки на базовый класс, должны иметь возможность использовать объекты производных классов, не зная об этом»_
> Robert C. Martin, _Agile Software Development_ (2002)

Иными словами, **наследник должен дополнять поведение родительского класса, а не подменять его**. Клиентский код, работающий с базовым типом, не должен «узнать», что на самом деле получил подтип, через неправильное поведение или исключения

### Простое объяснение «на пальцах»

Представьте, что у вас есть пульт управления универсальным роботом. Пульт обещает, что любое устройство с логотипом «Робот» умеет:
- Включаться
- Выключаться
- Выполнять команду «Принеси кофе»

Вы заказываете новую модель робота-официанта «Восьмёрка» с тем же логотипом. Вы подключаете пульт и нажимаете «Принеси кофе». Но робот внезапно говорит: «Не могу, у меня нет рук, я же робот-пылесос». Или ещё хуже: «Кофе подан» - и опрокидывает вам на колени кипяток

Такой робот **нарушает контракт** базового типа «Робот». Вы больше не можете доверять пульту. Приходится проверять, что за робот перед вами, и писать `if (robot is RobotV8) ... else ...`. Пульт перестаёт быть универсальным

**LSP требует, чтобы подтип вёл себя так, как ожидается от базового типа**

Если в документации к базовому классу `Bird` сказано «метод `Fly()` перемещает птицу в воздухе», то подкласс `Penguin` не должен «ломать» это ожидание - либо вообще не переопределять `Fly()` (если птица не летает, возможно, иерархия спроектирована неверно), либо выбрасывать исключение

Это означает, что **наследование - это не трюк для переиспользования кода, а семантический контракт**. Производный класс должен соблюдать все правила, установленные базовым

### Реальный пример
#### Классическая ловушка: Прямоугольник и Квадрат

Этот пример - «золотой стандарт» демонстрации нарушения LSP. Рассмотрим класс `Rectangle` (Прямоугольник):

```csharp
public class Rectangle
{
    public virtual int Width { get; set; }
    public virtual int Height { get; set; }
	
    public int CalculateArea()
    {
        return Width * Height;
    }
}
```

Теперь определим класс `Square` (Квадрат), который _с точки зрения геометрии_ является прямоугольником. Наивная реализация через наследование:

```csharp
public class Square : Rectangle
{
    public override int Width
    {
        set { base.Width = base.Height = value; }
    }
	
    public override int Height
    {
        set { base.Width = base.Height = value; }
    }
}
```

На первый взгляд, всё красиво: мы не даём установить ширину и высоту по-отдельности - они всегда одинаковы. Но посмотрим на клиентский код, который пишет юнит-тест для класса `Rectangle`:

```csharp
public void ResizeRectangle(Rectangle rect)
{
    rect.Width = 10;
    rect.Height = 5;
    Debug.Assert(rect.CalculateArea() == 50); // Ожидаем 10 * 5 = 50
}
```

Этот метод ожидает, что после изменения ширины и высоты площадь будет равна их произведению. Но если мы передадим в него `Square`, то получим:

```csharp
Rectangle square = new Square();
ResizeRectangle(square);
// square.Width = 10  ->  Width = 10, Height = 10
// square.Height = 5  ->  Width = 5,  Height = 5
// Assert: 5 * 5 = 25, но ожидалось 50 — ТЕСТ ПРОВАЛЕН
```

**Квадрат нарушает ожидания, установленные для Прямоугольника.** Разработчик функции `ResizeRectangle` полагался на постусловие «площадь = ширина × высота после раздельного присваивания». Квадрат это постусловие ломает

Формально: у `Rectangle` есть инвариант «ширину и высоту можно менять независимо». Квадрат этот инвариант не поддерживает. Следовательно, с точки зрения LSP, `Square` **не является корректным подтипом** `Rectangle`, и их иерархия должна быть пересмотрена (например, через общий интерфейс `IShape` с методом `CalculateArea()`)

#### Практический пример: Хранилище данных с разной семантикой

Допустим, у нас есть интерфейс хранилища заказов:

```csharp
public interface IOrderRepository
{
    void Add(Order order);
    Order GetById(int id);
    void Delete(int id);
}
```

И две реализации: `SqlOrderRepository` (хранит заказы в постоянной базе данных SQL) и `ReadOnlyOrderRepository` (возвращает данные из кэша или архива только для чтения)

Плохая реализация `ReadOnlyOrderRepository`:

```csharp
public class ReadOnlyOrderRepository : IOrderRepository
{
    public void Add(Order order)
    {
        throw new NotSupportedException("Репозиторий только для чтения");
    }
	
    public Order GetById(int id)
    {
        // Возвращаем данные из read-only кэша
        return _cache[id];
    }
	
    public void Delete(int id)
    {
        throw new NotSupportedException("Репозиторий только для чтения");
    }
}
```

Клиентский код, который проводит пакетную операцию переноса заказов:

```csharp
public void MigrateOrders(IOrderRepository source, IOrderRepository target)
{
    var orders = source.GetAll(); // предположим, есть GetAll
    foreach (var order in orders)
    {
        target.Add(order);
        source.Delete(order.Id);
    }
}
```

Если в качестве `target` передать `ReadOnlyOrderRepository`, метод `MigrateOrders` выбросит исключение. Это прямое нарушение LSP: подтип не выполняет контракт базового типа, клиентский код вынужден знать о конкретной реализации

**Исправление:** Интерфейс `IOrderRepository` был спроектирован некорректно (нарушение ISP - Interface Segregation Principle). Его следует разделить:

```csharp
public interface IReadableOrderRepository
{
    Order GetById(int id);
    IEnumerable<Order> GetAll();
}

public interface IWritableOrderRepository
{
    void Add(Order order);
    void Delete(int id);
}

public interface IOrderRepository : IReadableOrderRepository, IWritableOrderRepository
{
    // Полный доступ
}
```

Теперь `ReadOnlyOrderRepository` реализует только `IReadableOrderRepository`, а `SqlOrderRepository` — полный `IOrderRepository`. Метод `MigrateOrders` будет ожидать `IWritableOrderRepository` для `target` и соответствующий read-only для `source`. Контракты соблюдены, подстановка безопасна

### Зачем нужен принцип подстановки Лисков?

1. **Предсказуемость поведения**
   Клиентский код, опирающийся на базовый тип, может быть уверен, что любой производный тип будет работать согласно контракту. Это сокращает количество проверок `is`/`as` и условных конструкций, делая архитектуру чище
2. **Безопасность расширения**
   Без LSP невозможно реализовать принцип открытости/закрытости (OCP). Если новый подкласс может сломать существующий клиентский код, то система не является по-настоящему закрытой для модификации
3. **Снижение связанности**
   Клиенты зависят только от абстракции (базового типа) и не должны думать о конкретных реализациях. LSP гарантирует, что абстракция остаётся надёжной
4. **Упрощение тестирования**
   Юнит-тесты, написанные для базового типа, можно запускать для всех его подтипов (при правильной организации тестов), и они не должны падать из-за нарушения контракта

### Как применять LSP на практике?

1. **Определяйте контракты в терминах предусловий и постусловий**
	- **Предусловия** - что ожидает метод от вызывающего кода (например, аргумент не равен `null`). Подтип может **ослаблять** предусловия (т.е. принимать более широкий диапазон входных данных), но не усиливать их
	- **Постусловия** - что метод гарантирует на выходе (например, возвращаемое значение всегда положительное). Подтип может **усиливать** постусловия (давать более строгие гарантии), но не ослаблять их
	- **Инварианты** - условия, которые всегда истинны для объектов класса. Подтип должен сохранять инварианты базового класса
	В примере с `Rectangle` инвариант «ширину и высоту можно устанавливать независимо» был нарушен `Square`, который добавил связь между ними
2. **Избегайте наследования только ради повторного использования кода**
   Если класс наследует другой класс исключительно ради пары методов, но при этом нарушает его семантику - это тревожный сигнал. Используйте композицию или выделите общий интерфейс
3. **Обращайте внимание на исключения NotSupportedException и NotImplementedException**
   Если производный класс выбрасывает такие исключения для методов базового интерфейса, это красный флаг нарушения LSP (и ISP). Это говорит о том, что абстракция слишком «толстая»
4. **Предпочитайте композицию наследованию**
   Если поведение не полностью соответствует отношению «является» (is-a), используйте вложенные объекты-стратегии (паттерн «Стратегия») или обёртки (паттерн «Декоратор»), чтобы не ломать иерархию
5. **Следуйте правилу «не удивляй клиента»**
   Мысленно проверьте: если вы передадите подтип в метод, который знает только о базовом типе, будет ли результат ожидаемым? Если вы чувствуете, что нужен `if (obj is DerivedType d) ...` - иерархия нарушает LSP
6. **Используйте тесты контрактов**
   Создавайте абстрактные тестовые классы для базового интерфейса и запускайте их для всех реализаций. Так вы сразу обнаружите нарушения контракта новым подклассом

### Связь с другими принципами SOLID

- **Единственной ответственности (SRP)**: LSP помогает соблюдать SRP, потому что корректная иерархия с чёткими контрактами подталкивает к тому, чтобы каждый подтип имел одну причину для изменения, не ломая другие
- **Открытости/закрытости (OCP)**: LSP - необходимое условие для OCP. Если подтипы могут нарушать ожидания, систему нельзя расширять добавлением новых подтипов без модификации клиентского кода
- **Разделения интерфейса (ISP)**: Часто нарушение LSP является следствием нарушения ISP. «Толстые» интерфейсы провоцируют создание реализаций, которые не могут выполнить все методы, и тогда выбрасываются исключения
- **Инверсии зависимостей (DIP)**: DIP призывает зависеть от абстракций. LSP гарантирует, что эти абстракции работают корректно при передаче любых реализаций

### Основные источники

- **Liskov B.** _Data Abstraction and Hierarchy_ (1987) - оригинальная работа, положившая начало принципу
- **Martin R. C.** _Agile Software Development: Principles, Patterns, and Practices_ (2002) - формулировка для ООП, примеры нарушения и исправления LSP
- **Martin R. C.** _Clean Architecture_ (2017) - обсуждение LSP в контексте архитектурных границ
- **Gamma E., Helm R., Johnson R., Vlissides J.** _Design Patterns_ (1994) - множество паттернов, помогающих соблюдать LSP через композицию и делегирование
