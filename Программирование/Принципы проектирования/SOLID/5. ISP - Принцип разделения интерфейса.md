## Interface Segregation Principle (ISP)
### Формулировка принципа

Принцип разделения интерфейса был сформулирован Робертом Мартином в 1995 году, когда он консультировал компанию Xerox по перепроектированию программного обеспечения для принтеров. В оригинале он звучит так:

> _«Клиенты не должны зависеть от методов, которые они не используют»_
> Robert C. Martin, _The Interface Segregation Principle_ (1996)

В книге _Agile Software Development_ (2002) Мартин уточняет:

> _«Много специализированных интерфейсов лучше, чем один интерфейс общего назначения»_

Принцип предостерегает от создания «толстых» интерфейсов - таких, которые содержат слишком много методов и вынуждают реализующие их классы поддерживать функциональность, которая им не нужна. Вместо этого интерфейсы следует разделять на более мелкие, узкоспециализированные, чтобы каждый клиентский код зависел только от тех методов, которые ему действительно необходимы

### Простое объяснение «на пальцах»

Представьте, что вы покупаете швейцарский армейский нож, в котором есть лезвие, ножницы, открывалка для бутылок, напильник, отвёртка и USB-флешка. Вы - повар, и вам нужен только нож. Но продавец говорит: «Берите всё сразу, нам так проще продавать». Теперь, если производитель вспомнит о дефекте во флешке и отзовёт товар для замены, вы тоже должны будете отдать свой нож в ремонт, хотя флешкой никогда не пользовались. Или если вы попытаетесь взять его в самолёт, служба безопасности конфискует всё устройство из-за лезвия, хотя вам нужна была только отвёртка

С точки зрения интерфейсов, такой «швейцарский нож» - это **толстый интерфейс**. Он заставляет зависеть от ненужных деталей, увеличивает риск изменений и усложняет использование

В мире объектов: если у вас есть интерфейс `IMultiFunctionDevice`, включающий методы `Print()`, `Scan()`, `Fax()`, и вы создаёте простой домашний принтер, в котором нет сканера и факса, вам всё равно придётся реализовывать все три метода (или выбрасывать `NotSupportedException`). Клиент, который знает только об `IMultiFunctionDevice`, может вызвать `Scan()` на вашем принтере и сломаться. ISP требует разбить этот интерфейс на `IPrinter`, `IScanner`, `IFax`, чтобы каждый класс реализовывал только то, что ему нужно

### Реальный пример
#### Нарушение ISP: «толстый» интерфейс

Предположим, мы проектируем систему для работы с документами. Изначально у нас есть один интерфейс `IDocumentProcessor`, который умеет всё:

```csharp
public interface IDocumentProcessor
{
    void Open(string filePath);
    void Save(string filePath);
    void Print();
    void Scan();
    void Fax(string phoneNumber);
}
```

Класс `PdfDocumentProcessor` реализует открытие, сохранение и печать PDF-файлов, но не имеет возможности сканировать или отправлять факс:

```csharp
public class PdfDocumentProcessor : IDocumentProcessor
{
    public void Open(string filePath)
    {
        Console.WriteLine($"Открываем PDF: {filePath}");
    }
	
    public void Save(string filePath)
    {
        Console.WriteLine($"Сохраняем PDF: {filePath}");
    }
	
    public void Print()
    {
        Console.WriteLine("Печатаем PDF");
    }
	
    public void Scan()
    {
        throw new NotSupportedException("PDF-процессор не поддерживает сканирование");
    }
	
    public void Fax(string phoneNumber)
    {
        throw new NotSupportedException("PDF-процессор не поддерживает отправку факсов");
    }
}
```

Когда клиентский код, работающий с факсом, получает объект через интерфейс `IDocumentProcessor`, он ожидает, что метод `Fax()` работает:

```csharp
public void SendDocumentByFax(IDocumentProcessor processor, string phoneNumber)
{
    // ... подготовка ...
    processor.Fax(phoneNumber); // Здесь упадёт с NotSupportedException
}
```

Это классический «жирный» интерфейс. Он заставляет нас реализовывать методы, которые не имеют смысла для конкретного класса, и создаёт ложное чувство безопасности у клиентов. Также при изменении любого из методов интерфейса (например, добавлении нового параметра в `Scan()`) все реализации, даже те, что не поддерживают сканирование, должны быть перекомпилированы и изменены

#### Применяем ISP: разделяем интерфейсы

Следуя принципу разделения интерфейса, мы разбиваем `IDocumentProcessor` на несколько специализированных:

```csharp
public interface IOpener
{
    void Open(string filePath);
}

public interface ISaver
{
    void Save(string filePath);
}

public interface IPrinter
{
    void Print();
}

public interface IScanner
{
    void Scan();
}

public interface IFax
{
    void Fax(string phoneNumber);
}
```

Теперь класс `PdfDocumentProcessor` реализует только те интерфейсы, которые ему нужны:

```csharp
public class PdfDocumentProcessor : IOpener, ISaver, IPrinter
{
    public void Open(string filePath)
    {
        Console.WriteLine($"Открываем PDF: {filePath}");
    }
	
    public void Save(string filePath)
    {
        Console.WriteLine($"Сохраняем PDF: {filePath}");
    }
	
    public void Print()
    {
        Console.WriteLine("Печатаем PDF");
    }
}
```

Для многофункционального устройства (МФУ) мы можем либо реализовать все интерфейсы в одном классе, либо скомбинировать их через композицию:

```csharp
public class MultiFunctionDevice : IOpener, ISaver, IPrinter, IScanner, IFax
{
    // Реальная реализация всех методов
}
```

Клиентский код, работающий только с печатью, теперь зависит только от `IPrinter`:

```csharp
public class PrintService
{
    private readonly IPrinter _printer;
	
    public PrintService(IPrinter printer)
    {
        _printer = printer;
    }
	
    public void PrintDocument()
    {
        _printer.Print();
    }
}
```

Теперь:
- `PdfDocumentProcessor` не содержит бессмысленных методов
- Клиент `PrintService` не знает ничего о сканировании и факсе
- При изменении интерфейса `IFax` (добавление нового метода) класс `PdfDocumentProcessor` не затрагивается, так как он от него не зависит
- Код стал более устойчивым к изменениям, лучше читаемым и легче тестируемым

### Зачем нужен принцип разделения интерфейса?

1. **Избежание зависимостей от «мусорных» методов**
   Когда класс реализует интерфейс с методами, которые не использует, он всё равно вынужден компилироваться, линковаться и потенциально обрабатывать изменения в этих методах. ISP устраняет эту лишнюю связанность
2. **Повышение надёжности**
   Клиенты, зависящие от «жирного» интерфейса, могут вызвать метод, который на конкретной реализации выбросит `NotSupportedException`. Это ошибка времени выполнения, которую трудно предвидеть. ISP делает такие ошибки принципиально невозможными
3. **Упрощение поддержки и расширения**
   При изменении требований достаточно модифицировать или создать новый узкий интерфейс, не затрагивая несвязанные части системы. Многофункциональные классы могут формироваться путём композиции маленьких интерфейсов, а не монолитным наследованием
4. **Улучшение читаемости кода**
   Глядя на список интерфейсов класса (`IOpener, ISaver, IPrinter`), мы сразу понимаем его возможности. Не нужно проваливаться в реализацию и искать, какие методы рабочие, а какие бросают исключения
5. **Снижение связанности модулей**
   Зависимость от маленьких интерфейсов позволяет легко заменять одну реализацию другой, не требуя от замены всей функциональности. Это упрощает модульное тестирование (mocking) и применение паттерна «Декоратор»

### Как применять ISP на практике?

1. **Ищите жирные интерфейсы**
	Признаки проблемного интерфейса:
	- Количество методов больше, чем нужно большинству его реализаций
	- Одна или несколько реализаций выбрасывают `NotSupportedException` или `NotImplementedException`
	- При изменении интерфейса (добавлении метода) приходится «дотрагиваться» до многих несвязанных классов
	- Название интерфейса содержит слова «And» или «Or» (например, `IPrintAndScan`), что указывает на объединение несвязанной функциональности
2. **Применяйте разделение на основе ролей**
   Подумайте, какие роли (клиенты) будут использовать ваш класс. Каждая роль должна видеть только свой интерфейс. Например, для заказа могут быть интерфейсы `IOrderReader` (для отображения), `IOrderProcessor` (для обработки), `IInvoiceGenerator` (для выставления счёта)
3. **Используйте наследование интерфейсов для комбинации**
   Если в определённом контексте нужен набор из нескольких узких интерфейсов, можно создать объединяющий интерфейс, но только при реальной необходимости. Это лучше, чем один жирный интерфейс с кучей методов, потому что клиенты могут выбирать, от какого из базовых интерфейсов им зависеть
4. **Следуйте принципу при проектировании API**
   При создании библиотек или публичных API особенно важно не навязывать пользователям реализацию ненужных методов. Лучше предоставить пять маленьких интерфейсов, чем один большой
5. **Не доводите до абсурда**
   Как и с другими принципами, излишнее дробление может привести к появлению тысяч однострочных интерфейсов, в которых невозможно ориентироваться. Останавливайтесь на том уровне гранулярности, когда каждый интерфейс инкапсулирует **целостную, атомарную** единицу поведения, определяемую требованиями клиента. Интерфейс `IPrinter` с одним методом `Print` является примером хорошей гранулярности. Интерфейс `IOpenable` с методом `Open` - тоже. А вот `ISetName` с одним методом `SetName`, возможно, уже излишен, если только это не критически важная абстракция

### Связь с другими принципами SOLID

- **Единственной ответственности (SRP)**: ISP помогает соблюдать SRP на уровне интерфейсов. Класс, реализующий несколько узких интерфейсов, естественным образом имеет несколько ответственностей, но каждая из них изолирована и может быть вынесена в отдельный объект при необходимости
- **Открытости/закрытости (OCP)**: ISP облегчает соблюдение OCP, потому что расширение системы новым поведением часто сводится к добавлению нового узкого интерфейса и его реализации, не затрагивая существующие интерфейсы и их клиентов
- **Подстановки Лисков (LSP)**: Нарушение LSP часто является следствием жирного интерфейса. Если бы интерфейс был разделён, классы, не способные выполнить некоторые методы, просто не реализовывали бы «лишний» интерфейс, и подстановка была бы безопасной
- **Инверсии зависимостей (DIP)**: ISP усиливает DIP, потому что зависимости от узких абстракций (интерфейсов) легче инвертировать и подменять, чем зависимости от одного всеобъемлющего интерфейса, который тянет за собой целый ворох ненужных деталей

### Основные источники

- **Martin R. C.** _The Interface Segregation Principle_ (1996) - оригинальная статья, описывающая проблему «жирных» интерфейсов на примере программного обеспечения Xerox
- **Martin R. C.** _Agile Software Development: Principles, Patterns, and Practices_ (2002) - глава 12, детальный разбор ISP с примерами на C++ и Java
- **Martin R. C.** _Clean Architecture_ (2017) - обсуждение ISP в контексте архитектурных границ и управления зависимостями
- **Microsoft .NET Documentation** - Best practices for interface design in .NET
