#SOLID #OCP

## Open/Closed Principle (OCP)
### Формулировка принципа

> _«Программные сущности (классы, модули, функции и т. д.) должны быть открыты для расширения, но закрыты для модификации»_
> Бертран Мейер, _Object-Oriented Software Construction_ (1988)

Роберт Мартин адаптировал этот принцип для объектно-ориентированного проектирования, сформулировав его как:

> _«Вы должны иметь возможность расширять поведение модуля, не изменяя его исходного кода»_

Иными словами, когда требования меняются (а они меняются всегда), мы должны добавлять новую функциональность через **написание нового кода**, а не через переписывание старого, уже отлаженного и протестированного

### Простое объяснение «на пальцах»

Представьте, что вы купили навороченный системный блок. Через год вы решили увеличить объём оперативной памяти. Если для этого нужно разбирать блок питания, перепаивать дорожки на материнской плате и переустанавливать процессор - это «закрытая» конструкция. Хороший же корпус предоставляет стандартные слоты: открываете крышку, вставляете новую планку памяти в свободный разъём - и всё работает. Системный блок **закрыт для модификации** (вы не трогаете материнскую плату), но **открыт для расширения** (добавление новых планок памяти)

Другой пример - USB-порт. Компьютер не нужно модифицировать, чтобы подключить новое устройство. Производитель клавиатуры, мыши или внешнего диска просто создаёт устройство, совместимое со стандартным интерфейсом. Сам порт и операционная система не меняются

В программировании аналогом USB-разъёма являются **абстракции** (интерфейсы, базовые классы). Они определяют контракт, не меняющийся со временем. Новая функциональность добавляется путём создания новых реализаций этого контракта

### Реальный пример
#### Нарушение OCP: жёстко закодированная логика

Допустим, у нас есть система расчёта премий для сотрудников. Изначально премия рассчитывается только для штатных сотрудников как процент от оклада. Код выглядит так:

```csharp
public class Employee
{
    public string Type { get; set; } // "Regular" или "Contractor"
    public string Name { get; set; }
    public decimal Salary { get; set; }
    public decimal HourlyRate { get; set; }
    public int HoursWorked { get; set; }
}

public class BonusCalculator
{
    public decimal CalculateBonus(Employee employee)
    {
        if (employee.Type == "Regular")
        {
            // Премия для штатных: 10% от оклада
            return employee.Salary * 0.10m;
        }
        else if (employee.Type == "Contractor")
        {
            // Премия для контрактников: фиксированная ставка за час
            return employee.HourlyRate * 5; // 5 часов премиальных
        }
        else
        {
            return 0;
        }
    }
}
```

Когда компания решает ввести третий тип сотрудников - «Внештатные с почасовой оплатой на проектной основе» - нам придётся **модифицировать** класс `BonusCalculator`: добавить ещё одно условие `else if`. Более того, при появлении каждого нового типа сотрудников мы вынуждены лезть в этот же класс, рискуя сломать уже работающую логику

Это прямое нарушение OCP: модуль не закрыт для модификации

#### Применяем OCP: полиморфизм через абстракцию

Следуя OCP, мы должны отделить изменяющееся поведение (расчёт премии) за абстракцию и предоставить разные реализации для разных типов сотрудников. При появлении нового типа мы **добавим новый класс**, не трогая существующие

**Шаг 1.** Определяем интерфейс (контракт):

```csharp
public interface IBonusCalculator
{
    decimal CalculateBonus(Employee employee);
}
```

**Шаг 2.** Для каждого типа сотрудника создаём отдельную реализацию:

```csharp
public class RegularEmployeeBonusCalculator : IBonusCalculator
{
    public decimal CalculateBonus(Employee employee)
    {
        return employee.Salary * 0.10m;
    }
}

public class ContractorBonusCalculator : IBonusCalculator
{
    public decimal CalculateBonus(Employee employee)
    {
        return employee.HourlyRate * 5; // 5 часов премиальных
    }
}
```

**Шаг 3.** Модифицируем класс `Employee` так, чтобы он принимал стратегию расчёта премии:

```csharp
public class Employee
{
    public string Name { get; set; }
    public decimal Salary { get; set; }
    public decimal HourlyRate { get; set; }
    public int HoursWorked { get; set; }
    
    // Внедрение стратегии расчёта премии (инверсия зависимостей)
    private readonly IBonusCalculator _bonusCalculator;
	
    public Employee(IBonusCalculator bonusCalculator)
    {
        _bonusCalculator = bonusCalculator;
    }
	
    public decimal GetBonus()
    {
        return _bonusCalculator.CalculateBonus(this);
    }
}
```

Теперь код, использующий сотрудников, выглядит так:

```csharp
var regularEmployee = new Employee(new RegularEmployeeBonusCalculator())
{
    Name = "Анна",
    Salary = 100000
};

var contractor = new Employee(new ContractorBonusCalculator())
{
    Name = "Петр",
    HourlyRate = 1500
};

Console.WriteLine($"Премия Анны: {regularEmployee.GetBonus()}");
Console.WriteLine($"Премия Петра: {contractor.GetBonus()}");
```

Когда бизнес попросит добавить новый тип сотрудника - например, «Стажёр» с фиксированной премией, мы просто создадим новый класс:

```csharp
public class InternBonusCalculator : IBonusCalculator
{
    public decimal CalculateBonus(Employee employee)
    {
        return 5000; // фиксированная премия
    }
}
```

Классы `Employee`, `RegularEmployeeBonusCalculator`, `ContractorBonusCalculator` останутся **нетронутыми**. Система открыта для расширения (через добавление нового класса) и закрыта для модификации (существующий код не менялся)

### Зачем нужен принцип открытости/закрытости?

1. **Предотвращение регрессии**
   Каждое изменение существующего кода несёт риск внесения ошибок. Если мы не трогаем старый код, мы не ломаем то, что уже работает и протестировано
2. **Упрощение тестирования**
   Новые классы изолированы и могут быть протестированы независимо. Нет необходимости переписывать или перезапускать тесты для старой функциональности
3. **Повышение гибкости архитектуры**
   Система, спроектированная по OCP, легко адаптируется к новым требованиям. Это особенно важно в долгоживущих проектах и продуктах с частыми релизами
4. **Улучшение масштабируемости команд**
   Разные команды могут параллельно разрабатывать новые «расширения» (новые классы), не мешая друг другу и не наступая на общий код
5. **Соблюдение принципа единственной ответственности**
   OCP естественным образом следует из SRP: когда каждый класс отвечает за одну задачу, его проще оставить закрытым для изменений, а новые задачи выделять в новые классы

### Как применять OCP на практике?

1. **Определите оси изменений**
   Проанализируйте, какие части системы будут меняться чаще всего. Обычно это бизнес-правила, алгоритмы расчётов, форматы вывода данных, способы взаимодействия с внешними системами. Именно их стоит изолировать за абстракциями
2. **Программируйте на уровне абстракций, а не реализации**
   Вместо конкретных классов используйте интерфейсы и базовые классы. Внедряйте зависимости через конструктор (Dependency Injection). Это фундамент для OCP
3. **Используйте композицию вместо наследования**
   Предпочтительнее делегировать изменяемое поведение отдельным объектам-стратегиям (паттерн «Стратегия»), а не создавать громоздкие иерархии наследования
4. **Применяйте проверенные паттерны проектирования**
	Многие паттерны напрямую реализуют OCP:
	- **Стратегия (Strategy)** - инкапсулирует семейство алгоритмов (как в примере с премиями)
	- **Декоратор (Decorator)** - расширяет поведение объекта без изменения его кода
	- **Шаблонный метод (Template Method)** - определяет скелет алгоритма, позволяя подклассам переопределять отдельные шаги
	- **Компоновщик (Composite)** - позволяет единообразно обращаться с единичными и составными объектами
	- **Цепочка обязанностей (Chain of Responsibility)** - расширяет обработку запросов добавлением новых обработчиков
5. **Не пытайтесь предугадать все будущие расширения**
   Слепое следование OCP может привести к излишней сложности (преждевременной абстракции). Роберт Мартин предостерегает: защищаться нужно лишь от тех изменений, которые **вероятны**, основываясь на понимании предметной области. Если вы не знаете, что может измениться, не усложняйте код заранее. Однако как только появляется второе похожее изменение (например, второй тип сотрудника), смело проводите рефакторинг к абстракции
6. **Рефакторинг существующего кода**
   Если вы видите цепочки `if-else` или `switch`, проверяющие тип объекта для выбора поведения - это кандидат на применение OCP. Вынесите каждую ветку в отдельный класс, реализующий общий интерфейс. Делайте это маленькими шагами, под прикрытием тестов

### Связь с другими принципами SOLID

- **Единственной ответственности (SRP)**: OCP является логическим продолжением SRP. Когда класс имеет только одну ответственность, его редко нужно менять. Расширение же осуществляется за счёт новых классов, берущих на себя новые ответственности
- **Подстановки Лисков (LSP)**: OCP требует, чтобы новые расширения были корректны по отношению к базовым абстракциям. Без LSP невозможно безопасно подставлять новые реализации, и OCP теряет смысл
- **Разделения интерфейса (ISP)**: Чем уже и специализированнее интерфейс, тем легче написать его новую реализацию, то есть тем проще расширять систему без модификации
- **Инверсии зависимостей (DIP)**: OCP опирается на DIP - мы зависим от абстракций (интерфейс `IBonusCalculator`), а не от конкретных классов. Это и есть механизм «закрытости»
