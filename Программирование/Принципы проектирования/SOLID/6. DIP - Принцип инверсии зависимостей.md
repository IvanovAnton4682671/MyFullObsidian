## Dependency Inversion Principle (DIP)
### Формулировка принципа

Принцип инверсии зависимостей был сформулирован Робертом Мартином в 1996 году как стратегия борьбы с жёсткой связанностью модулей. В оригинале он состоит из двух утверждений:

> _«А. Высокоуровневые модули не должны зависеть от низкоуровневых модулей. И те, и другие должны зависеть от абстракций
> Б. Абстракции не должны зависеть от деталей. Детали должны зависеть от абстракций»_
> Robert C. Martin, _The Dependency Inversion Principle_ (1996)

В книге _Agile Software Development_ (2002) Мартин уточняет:

> _«Зависимости внутри системы должны быть направлены на абстракции, а не на конкретные реализации»_

Ключевое слово здесь - **«инверсия»**. Традиционно высокоуровневый код (содержащий бизнес-логику) зависел от низкоуровневого (работа с базой данных, файлами, сетью). DIP переворачивает это отношение: и высокоуровневый, и низкоуровневый модули зависят от общей абстракции, задаваемой высокоуровневой политикой. В результате детали становятся подчинёнными бизнес-правилам, а не наоборот

### Простое объяснение «на пальцах»

Представьте настольную лампу, которая намертво припаяна к одной-единственной электрической розетке в стене. Если розетка сломается или вы захотите переставить лампу в другой угол, вам придётся демонтировать проводку и перепаивать соединения. Это жёсткая зависимость от конкретной розетки

В реальном мире производители ламп и строители договорились о стандартном интерфейсе: **штепсельная вилка + розетка стандарта CEE 7/4 (Schuko)**. Лампа ничего не знает о том, откуда берётся электричество - от электростанции, солнечной батареи или генератора. Ей важно только одно: чтобы розетка предоставляла переменный ток 230 В 50 Гц. Это и есть **абстракция**. Лампа (высокоуровневый модуль) зависит от абстрактного «источника питания», а не от конкретной электростанции (низкоуровневый модуль). И розетка тоже реализует эту абстракцию, подчиняясь стандарту, заданному потребностями лампы, а не наоборот

Это и есть инверсия зависимости: не лампа подстраивается под конкретную электростанцию, а электростанция и лампа оба подчиняются единому стандарту - абстракции, продиктованной высокоуровневым требованием «нужно питание 230 В»

В программировании аналогом розетки является **интерфейс**, задаваемый высокоуровневым кодом. Низкоуровневые классы реализуют этот интерфейс, «втыкаясь» в систему как вилка в розетку

### Реальный пример
#### Нарушение DIP: прямое использование низкоуровневого модуля

Рассмотрим приложение для отправки уведомлений. У нас есть высокоуровневый класс `NotificationService`, который отправляет уведомления пользователям по электронной почте. В первой версии он напрямую использует конкретный SMTP-клиент:

```csharp
// Низкоуровневый модуль — конкретная реализация отправки писем
public class SmtpEmailSender
{
    public void SendEmail(string to, string subject, string body)
    {
        // Подключение к SMTP-серверу, аутентификация, отправка
        Console.WriteLine($"Отправляем письмо через SMTP на {to}: {subject}");
    }
}

// Высокоуровневый модуль — бизнес-логика уведомлений
public class NotificationService
{
    private readonly SmtpEmailSender _emailSender;
	
    public NotificationService()
    {
        _emailSender = new SmtpEmailSender(); // Жёсткая зависимость
    }
	
    public void SendOrderConfirmation(string customerEmail, int orderId)
    {
        string subject = "Заказ подтверждён";
        string body = $"Ваш заказ #{orderId} принят в обработку.";
        _emailSender.SendEmail(customerEmail, subject, body);
    }
}
```

**Проблемы этого подхода:**

1. **Жёсткая связанность**
   `NotificationService` знает о конкретном классе `SmtpEmailSender`. Если мы захотим отправлять уведомления через SMS, Push или Telegram, придётся переписывать `NotificationService`
2. **Сложность тестирования**
   Чтобы протестировать бизнес-логику `NotificationService`, мы вынуждены поднимать реальный SMTP-сервер или городить сложные перехватчики на уровне сети. Юнит-тест становится интеграционным
3. **Нарушение потока зависимостей**
   Высокоуровневая политика (бизнес-правило «после заказа отправляется уведомление») зависит от низкоуровневой детали (SMTP-протокол). Если завтра SMTP-сервер изменит API, придётся править бизнес-логику

#### Применяем DIP: вводим абстракцию

Следуя DIP, мы должны избавиться от прямой зависимости `NotificationService` от `SmtpEmailSender`. Для этого мы вводим абстракцию, которая принадлежит **высокоуровневому модулю** - интерфейс `IMessageSender`, определяющий операцию «отправить сообщение» в терминах бизнес-логики:

```csharp
// Абстракция, определённая высокоуровневым модулем
public interface IMessageSender
{
    void SendMessage(string recipient, string subject, string body);
}
```

Теперь `NotificationService` зависит только от этой абстракции:

```csharp
public class NotificationService
{
    private readonly IMessageSender _messageSender;
	
    // Внедрение зависимости через конструктор (Dependency Injection)
    public NotificationService(IMessageSender messageSender)
    {
        _messageSender = messageSender;
    }
	
    public void SendOrderConfirmation(string customerEmail, int orderId)
    {
        string subject = "Заказ подтверждён";
        string body = $"Ваш заказ #{orderId} принят в обработку.";
        _messageSender.SendMessage(customerEmail, subject, body);
    }
}
```

`SmtpEmailSender` теперь - всего лишь одна из реализаций абстракции, подчинённая ей:

```csharp
// Низкоуровневый модуль реализует абстракцию
public class SmtpEmailSender : IMessageSender
{
    public void SendMessage(string to, string subject, string body)
    {
        Console.WriteLine($"Отправляем письмо через SMTP на {to}: {subject}");
    }
}
```

Теперь мы можем легко добавить новые способы отправки, не меняя бизнес-логику:

```csharp
public class SmsSender : IMessageSender
{
    public void SendMessage(string phoneNumber, string subject, string body)
    {
        Console.WriteLine($"Отправляем SMS на {phoneNumber}: {body}");
    }
}

public class TelegramSender : IMessageSender
{
    public void SendMessage(string chatId, string subject, string body)
    {
        Console.WriteLine($"Отправляем Telegram сообщение в {chatId}: {body}");
    }
}
```

Сборка приложения на верхнем уровне (композиция объектов) связывает всё вместе:

```csharp
// Композиция корня приложения (Composition Root)
IMessageSender sender = new SmtpEmailSender(); // или new SmsSender(); 
var notificationService = new NotificationService(sender);
notificationService.SendOrderConfirmation("customer@example.com", 12345);
```

Направление зависимостей изменилось:
- **До:** `NotificationService` → `SmtpEmailSender` (стрелка шла от бизнес-логики к конкретной детали)
- **После:** `NotificationService` → `IMessageSender` ← `SmtpEmailSender`. Оба модуля зависят от абстракции, причём абстракция принадлежит высокоуровневому модулю (её диктует бизнес-логика «мне нужно отправить сообщение», а не SMTP-протокол)

Это и есть **инверсия зависимостей**

### Зачем нужен принцип инверсии зависимостей?

1. **Изоляция бизнес-логики от деталей**
   Бизнес-правила - самый ценный и стабильный актив программного обеспечения. DIP защищает их от вынужденных изменений при смене баз данных, UI-фреймворков, внешних сервисов или протоколов связи
2. **Лёгкость замены реализаций**
   Поскольку все модули зависят от интерфейсов, мы можем в любой момент подменить низкоуровневый модуль на другой - например, перейти с MS SQL на PostgreSQL или добавить кэширование через декоратор, не трогая высокоуровневый код
3. **Упрощение модульного тестирования**
   Зависимости, внедряемые через интерфейс, легко заменяются тестовыми дублёрами (mock, stub, fake). Это позволяет тестировать бизнес-логику изолированно, быстро и без подключения к реальной инфраструктуре
4. **Параллельная разработка**
   Команды могут договориться об интерфейсах и разрабатывать высокоуровневые и низкоуровневые модули независимо друг от друга, что ускоряет процесс
5. **Снижение связанности и повышение согласованности**
   DIP - самый мощный инструмент управления зависимостями в архитектуре. В сочетании с другими принципами SOLID он позволяет строить системы, которые легко поддерживать и расширять годами

### Как применять DIP на практике?

1. **Определите, где находятся «высокоуровневые политики»**
   Высокоуровневые модули - это те, что содержат бизнес-правила и принимают стратегические решения (как обрабатывать заказ, как рассчитывать скидку). Они не должны содержать кода работы с базой данных, отправки HTTP-запросов, форматирования вывода в консоль. Всё это - детали, от которых нужно абстрагироваться
2. **Владельцем интерфейса всегда является клиент**
   Интерфейс, объявляемый для инверсии зависимости, должен определяться в том же модуле/пакете, что и использующий его высокоуровневый код, а не там, где находится его реализация. Это ключевое правило: абстракция принадлежит потребителю, а не поставщику
3. **Используйте внедрение зависимостей (Dependency Injection, DI)**  
	DIP - это принцип, DI - паттерн, позволяющий его реализовать. Наиболее распространённые способы:
	- **Внедрение через конструктор** (рекомендуемый) - зависимости передаются при создании объекта
	- **Внедрение через свойство** - для опциональных зависимостей
	- **Внедрение через параметр метода** - если зависимость нужна только в одном методе
	Для управления зависимостями в крупных проектах применяются IoC-контейнеры (например, `Microsoft.Extensions.DependencyInjection`, Autofac, Ninject). Они автоматизируют создание объектов и разрешение цепочек зависимостей
4. **Не применяйте DIP к каждому классу**
   DIP нужен там, где вероятны изменения реализации. Не стоит создавать интерфейс для каждой тривиальной сущности (например, для простого DTO-класса). Мартин подчёркивает: применяйте DIP к «зависимостям, которые могут меняться по причинам, отличным от причин изменения использующего их модуля»
5. **Используйте паттерны «Адаптер» и «Шлюз» для внешних сервисов**
   Внешние библиотеки и API не реализуют ваши интерфейсы. Чтобы подчинить их вашей абстракции, создайте класс-адаптер, который оборачивает сторонний код и предоставляет ваш интерфейс. Это позволяет изолировать систему от чужих изменений
6. **Соблюдайте правило «стрелки зависимостей»**
   На диаграммах компонентов все стрелки зависимостей должны указывать **в сторону абстракции**, то есть в направлении, противоположном потоку управления. Конкретные классы могут зависеть только от интерфейсов (или других абстракций), но никогда - от других конкретных классов из внешних модулей

### Связь с другими принципами SOLID

- **Единственной ответственности (SRP)**: DIP естественным образом поддерживает SRP, так как разделение на абстракции и реализацию побуждает выделять классы с единственной ответственностью: бизнес-правила отдельно, инфраструктура отдельно
- **Открытости/закрытости (OCP)**: DIP - главный технический приём для достижения OCP. Система, построенная на абстракциях, открыта для расширения (добавления новых реализаций интерфейсов) и закрыта для модификации (бизнес-логика не меняется при смене деталей)
- **Подстановки Лисков (LSP)**: DIP предполагает, что любая реализация интерфейса может быть подставлена вместо другой. LSP гарантирует, что эта подстановка будет корректной и не сломает высокоуровневый код
- **Разделения интерфейса (ISP)**: DIP предписывает зависеть от абстракций, а ISP - делать эти абстракции узкими и специализированными. Вместе они образуют мощную комбинацию: мы зависим от маленьких интерфейсов, легко подменяем реализации и не страдаем от «толстых» зависимостей

### Основные источники

- **Martin R. C.** _The Dependency Inversion Principle_ (1996) - оригинальная статья, опубликованная в C++ Report
- **Martin R. C.** _Agile Software Development: Principles, Patterns, and Practices_ (2002) - глава 11, детальный разбор DIP с примерами
- **Martin R. C.** _Clean Architecture: A Craftsman's Guide to Software Structure and Design_ (2017) - роль DIP в построении «чистой архитектуры», графическое представление стрелок зависимостей
- **Gamma E., Helm R., Johnson R., Vlissides J.** _Design Patterns_ (1994) - паттерны «Абстрактная фабрика», «Стратегия», «Адаптер», тесно связанные с DIP
- **Seemann M.** _Dependency Injection in .NET_ (2011) - всестороннее руководство по внедрению зависимостей как реализации DIP на платформе .NET
