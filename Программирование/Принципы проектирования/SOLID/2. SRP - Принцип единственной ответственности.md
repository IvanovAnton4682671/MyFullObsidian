#SOLID #SRP
___
## Single Responsibility Principle (SRP)
### Формулировка принципа

> _«У класса должна быть только одна причина для изменения»_
> Роберт Мартин, _Agile Software Development: Principles, Patterns, and Practices_ (2002)

В более поздней трактовке из книги _«Чистая архитектура»_ (2017) Мартин уточняет:

> _«Модуль должен отвечать за одного и только за одного актора (заинтересованное лицо)»_

Таким образом, речь идёт не столько о том, что класс выполняет ровно одну задачу, сколько о том, что у него есть **единственный источник требований**, способный потребовать изменений. Класс инкапсулирует решение одной чётко очерченной проблемы для одной группы пользователей

### Простое объяснение «на пальцах»

Представьте, что вы наняли универсального специалиста, который должен:  
- разрабатывать дизайн интерфейсов
- писать серверный код
- администрировать базы данных
- ещё и чинить принтер в офисе

Пока компания маленькая, такой «мастер на все руки» справляется. Но как только любой из этих процессов усложняется (например, отдел дизайна переходит на новый инструмент), вам приходится переучивать всего специалиста целиком, рискуя сломать его навыки в других областях. Рано или поздно вы разделите эти обязанности на отдельных сотрудников: дизайнера, бэкенд-разработчика, администратора БД и системного администратора

Аналогично в коде: класс `Employee`, который сам считает себе зарплату, сам сохраняет себя в базу данных и сам генерирует отчёты, вынужден меняться по требованию трёх разных отделов. Бухгалтерия попросит поменять формулу расчёта - и это затронет код, используемый HR-отделом для отчётов. Изменение структуры базы данных сломает генерацию отчётов. Это создаёт ненужные зависимости и риск ошибок

Применяя **Single Responsibility Principle**, мы делим такой «универсальный» класс на несколько узкоспециализированных, каждый из которых обслуживает только **одного заказчика изменений**

### Реальный пример
#### Класс, нарушающий SRP

Рассмотрим типичный класс `Employee`, который одновременно управляет данными сотрудника, рассчитывает зарплату, сохраняет данные в базу данных и генерирует отчёт

```csharp
public class Employee
{
    public string Name { get; set; }
    public decimal HourlyRate { get; set; }
    public int HoursWorked { get; set; }
	
    // Рассчитывает зарплату (бизнес-логика)
    public decimal CalculatePay()
    {
        return HourlyRate * HoursWorked;
    }
	
    // Сохраняет сотрудника в БД (инфраструктура)
    public void SaveToDatabase()
    {
        string connectionString = "Server=.;Database=HR;...";
        // Код подключения к БД и SQL-запрос INSERT/UPDATE
    }
	
    // Генерирует строку отчёта (форматирование/представление)
    public string GenerateReport()
    {
        return $"Employee: {Name}, Earned: {CalculatePay():C}";
    }
}
```

**Чем это плохо?**
- Бухгалтерия просит изменить формулу расчёта оплаты (учёт сверхурочных, налогов) - придётся трогать класс, который отвечает и за сохранение, и за отчёты
- Администратор БД решает изменить структуру таблицы `Employees` - меняется метод `SaveToDatabase()`, и есть риск что-то сломать в `GenerateReport()`, хотя к отображению это не имеет отношения
- HR-отдел хочет изменить формат отчёта - заставляя перекомпилировать и перетестировать класс, затрагивающий хранение и расчёт зарплаты

Три совершенно разные причины для изменений живут в одном классе → **сильная связанность** (high coupling) и низкая согласованность (low cohesion)

#### Применяем SRP: выделяем ответственности

В соответствии с идеей SRP мы должны разделить этот класс так, чтобы каждый новый класс обслуживал только одного актора:
- **Бухгалтерию** интересует только расчёт зарплаты (`PayrollCalculator`)
- **Отдел кадров** - только данные о сотруднике и отчётность (`Employee`, `EmployeeReportGenerator`)
- **Инфраструктурная команда (или разработчики БД)** - способ сохранения (`EmployeeRepository`)

```csharp
// 1. Простая модель данных (DTO)
public class Employee
{
    public string Name { get; set; }
    public decimal HourlyRate { get; set; }
    public int HoursWorked { get; set; }
}

// 2. Расчёт зарплаты - ответственность перед бухгалтерией
public class PayrollCalculator
{
    public decimal Calculate(Employee employee)
    {
        // Позже здесь могут появиться налоги, сверхурочные, премии
        return employee.HourlyRate * employee.HoursWorked;
    }
}

// 3. Сохранение в базу данных — ответственность за хранение
public class EmployeeRepository
{
    public void Save(Employee employee)
    {
        // Код взаимодействия с БД
    }
}

// 4. Генерация отчётов — ответственность перед HR/аналитиками
public class EmployeeReportGenerator
{
    public string Generate(Employee employee, PayrollCalculator calculator)
    {
        decimal salary = calculator.Calculate(employee);
        return $"Employee: {employee.Name}, Earned: {salary:C}";
    }
}
```

Теперь, если бухгалтерия захочет внедрить учёт премий, мы поменяем только `PayrollCalculator`, не рискуя затронуть сохранение в БД или формат отчётов. Изменение схемы базы данных затронет лишь `EmployeeRepository`. Требования HR-отдела к отчётам - только `EmployeeReportGenerator`. Каждый класс имеет **одну причину для изменения**, то есть обслуживает одного актора

### Зачем нужен принцип единственной ответственности?

1. **Устойчивость к изменениям**
   Требования меняются всегда. Если у класса несколько причин для изменений, любая правка может затронуть неожиданные части системы. SRP изолирует области изменений, делая систему более предсказуемой
2. **Упрощение повторного использования**
   Класс, сфокусированный на одной задаче, легче переиспользовать в других контекстах. Например, `PayrollCalculator` можно применить в модуле формирования бюджета, не таща за собой логику отчётов или хранения
3. **Упрощение тестирования**
   Узкие классы легче тестировать юнит-тестами. Вам не нужно поднимать базу данных, чтобы проверить расчёт зарплаты, и не нужно заботиться о формулах, проверяя формат отчёта
4. **Повышение читаемости и прозрачности**
   Когда класс решает одну задачу, его назначение очевидно из названия и короткого кода. Это снижает когнитивную нагрузку на разработчика
5. **Облегчение командной разработки**
   Разные ответственности могут быть поручены разным разработчикам или командам с минимальными конфликтами при слиянии кода. Классы, принадлежащие разным «акторам», редко меняются одновременно

### Как применять SRP на практике?

1. **Определите акторов (источников требований)**
   Задайте вопрос: «Кто может попросить изменить этот класс?» Если вы можете назвать более одного отдела, пользовательской роли, бизнес-функции - вероятно, класс нарушает SRP
   Примеры акторов: бухгалтерия, HR, системные администраторы, отдел маркетинга, конечные пользователи web-интерфейса, мобильные клиенты
2. **Ищите признаки нарушения SRP:**
	- **Класс имеет слишком много полей и методов**, которые используются в разных, не связанных друг с другом сценариях
	- **Методы обращаются к разным слоям абстракции** - например, чистый расчёт и работа с файловой системой/сетью вперемешку
	- **Тестирование требует поднимать сложную инфраструктуру** (базы данных, веб-серверы) даже для проверки чистой логики
	- **Изменение одной детали ломает другие**, на первый взгляд не связанные функции
3. **Выделяйте ответственности последовательно**
   Начните с выявления наиболее очевидных «осей изменения»: бизнес-логика, сохранение, отображение, взаимодействие с внешними сервисами
   Создавайте отдельные классы, даже небольшие. Не бойтесь увеличить количество файлов - современные IDE с этим легко справляются
   Объединяйте классы по смыслу в пространства имён (namespaces) и сборки, чтобы структура проекта отражала зоны ответственности
4. **Остерегайтесь крайностей («SRP-фанатизм»)**
   Не стоит превращать каждый класс в однострочник с единственным методом. Гранулярность должна быть разумной: класс должен инкапсулировать **целостную ответственность**, а не просто «одно действие». Например, класс `PayrollCalculator` может содержать несколько методов расчёта (обычная зарплата, премия, сверхурочные) - это всё части одной ответственности «расчёт оплаты»
5. **Совмещайте с другими принципами**
   SRP тесно связан с **Interface Segregation Principle** (ISP): когда вы разделяете «толстый» интерфейс на несколько маленьких, вы часто неизбежно выделяете и ответственности. И с **Dependency Inversion Principle** (DIP): отделяя инфраструктурные детали (базу данных) от бизнес-логики, вы создаёте абстракции (интерфейс `IEmployeeRepository`), позволяющие подменять реализацию - это шаг зрелой архитектуры

### Связь с другими принципами SOLID

- **Открытости/закрытости (OCP)**: SRP закладывает основу для OCP, потому что класс, имеющий одну ответственность, легче расширить без модификации его исходного кода (добавлением новых классов, реализующих общий интерфейс)
- **Разделения интерфейса (ISP)**: Если интерфейс страдает от «раздутости», значит, класс, который его реализует, скорее всего, тоже нарушает SRP. Применение ISP часто приводит к соблюдению SRP
- **Инверсии зависимостей (DIP)**: SRP подталкивает к выделению инфраструктурных зависимостей в отдельные классы (репозитории, сервисы), что идеально согласуется с DIP — мы начинаем зависеть от абстракций, а не от конкретной реализации БД
